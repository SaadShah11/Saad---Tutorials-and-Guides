üöÄ Multi-Account Git Guide

A complete guide to managing multiple Git identities

This document walks you through the entire process of setting up your computer to handle multiple Git identities (e.g., a "personal" and "work" account) without them conflicting.

The setup is broken into two key concepts:

  1. Authentication (SSH Keys): This proves who you are to the Git server (like GitHub or GitLab). It controls your permission to push and pull code.
  2. Authorship (Git Config): This is the user.name and user.email metadata that gets stamped on your commits. It controls the "author" name that appears in the Git log.

We will configure your system so you never accidentally push work code from your personal account or vice-versa.

üßπ Part 0: Starting From a Clean Slate (Optional)

These steps will back up and remove your current global Git and SSH configurations. This is for users who want a completely fresh start.

‚ö†Ô∏è Warning: This is a destructive operation.

It will back up your old keys but will immediately revoke your computer's access to all remote servers until you complete the new setup.

macOS / Linux

  1. Open Terminal.
  2. Clear any loaded SSH keys from the agent: 
    ssh-add -D
  3. Back up and rename your existing SSH and Git config folders: 
    # Back up SSH
if [ -d ~/.ssh ]; then
  mv ~/.ssh ~/.ssh_backup
  echo "Backed up old SSH config to ~/.ssh_backup"
fi

# Back up Git Config
if [ -f ~/.gitconfig ]; then
  mv ~/.gitconfig ~/.gitconfig_backup
  echo "Backed up old Git config to ~/.gitconfig_backup"
fi
  4. Create a new, empty SSH directory with the correct permissions: 
    mkdir ~/.ssh
chmod 700 ~/.ssh

Windows

  1. Open PowerShell.
  2. Stop the ssh-agent and clear any loaded keys: 
    ssh-add -D
Stop-Service ssh-agent
  3. Back up and rename your existing SSH and Git config folders: 
    # Back up SSH (using $env:USERPROFILE, which is like ~)
if (Test-Path $env:USERPROFILE\.ssh) {
  Move-Item $env:USERPROFILE\.ssh $env:USERPROFILE\.ssh_backup
  Write-Host "Backed up old SSH config to .ssh_backup"
}

# Back up Git Config
if (Test-Path $env:USERPROFILE\.gitconfig) {
  Move-Item $env:USERPROFILE\.gitconfig $env:USERPROFILE\.gitconfig_backup
  Write-Host "Backed up old Git config to .gitconfig_backup"
}
  4. Create a new, empty SSH directory: 
    mkdir $env:USERPROFILE\.ssh

üîë Part 1: Setting Up SSH Keys (Authentication)

We will create a separate SSH key for each identity.

macOS / Linux

  1. Generate Keys: Open Terminal and run these commands.
    • Personal Key: 
      ssh-keygen -t ed25519 -C "your-personal-email@gmail.com" -f ~/.ssh/id_ed25519_personal
    • Work Key: 
      ssh-keygen -t ed25519 -C "your-work-email@company.com" -f ~/.ssh/id_ed25519_work
  2. Add Keys to ssh-agent: This securely stores your keys in the macOS Keychain. 
    ssh-add --apple-use-keychain ~/.ssh/id_ed25519_personal
ssh-add --apple-use-keychain ~/.ssh/id_ed25519_work
  3. Create SSH Config File:
    • Create and open the file: nano ~/.ssh/config
    • Paste the following content. These Host names are aliases (nicknames) you will use later. 
      # --- Personal Account (e.g., GitHub) ---
Host github.com-personal
    AddKeysToAgent no
    UseKeychain no
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_personal
    IdentitiesOnly yes

# --- Work Account (e.g., GitHub) ---
Host github.com-work
    AddKeysToAgent no
    UseKeychain no
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_work
    IdentitiesOnly yes
    
# --- Work Account (e.g., GitLab) ---
# Host gitlab.com-work
#     AddKeysToAgent no
#     UseKeychain no
#     HostName gitlab.com
#     User git
#     IdentityFile ~/.ssh/id_ed25519_work
#     IdentitiesOnly yes
    • Save and exit nano (Control+O, Enter, Control+X).
    • Set permissions for the file: chmod 600 ~/.ssh/config
  4. Add Public Keys to Git Providers:
    • Copy Personal Key: pbcopy < ~/.ssh/id_ed25519_personal.pub
      Go to your personal GitHub/GitLab account Settings > SSH and GPG keys > New SSH key and paste it.
    • Copy Work Key: pbcopy < ~/.ssh/id_ed25519_work.pub
      Go to your work GitHub/GitLab account Settings > SSH and GPG keys > New SSH key and paste it.

Windows (10/11)

  1. Start the OpenSSH Agent: Open PowerShell as an Administrator. 
    # Set the service to start automatically
Set-Service -Name ssh-agent -StartupType Automatic

# Start the service
Start-Service ssh-agent
  2. Generate Keys: Open a regular PowerShell window.
    • Personal Key: 
      ssh-keygen -t ed25519 -C "your-personal-email@gmail.com" -f "$HOME\.ssh\id_ed25519_personal"
    • Work Key: 
      ssh-keygen -t ed25519 -C "your-work-email@company.com" -f "$HOME\.ssh\id_ed25519_work"
  3. Add Keys to ssh-agent: 
    ssh-add "$HOME\.ssh\id_ed25519_personal"
ssh-add "$HOME\.ssh\id_ed25519_work"
  4. Create SSH Config File:
    • First, create an empty config file (without an extension): 
      New-Item -Path $HOME\.ssh\config -ItemType File -Force
    • Then open it in Notepad: 
      notepad $HOME\.ssh\config
      Important: If Notepad asks to create the file, click Yes. When saving, make sure to select "All Files (*.*)" in the "Save as type" dropdown to prevent Notepad from adding a .txt extension.
    • Paste the exact same content as in the macOS step: 
      # --- Personal Account (e.g., GitHub) ---
Host github.com-personal
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_personal
    IdentitiesOnly yes

# --- Work Account (e.g., GitHub) ---
Host github.com-work
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_work
    IdentitiesOnly yes
  5. Add Public Keys to Git Providers:
    • Copy Personal Key: 
      Get-Content $HOME\.ssh\id_ed25519_personal.pub | Set-Clipboard
      Go to your personal GitHub/GitLab account and paste it.
    • Copy Work Key: 
      Get-Content $HOME\.ssh\id_ed25519_work.pub | Set-Clipboard
      Go to your work GitHub/GitLab account and paste it.

üìù Part 2: Setting Up Git Config (Authorship)

This part is identical for both Windows and macOS.

Method A: The "Safety Net" (Per-Project Config)

This is the method you are currently using. By not having a global ~/.gitconfig file, Git will force you to set your identity for every new repository.

  • After you git clone or git init a project, cd into it.
  • If it's a personal project, run: 
    git config user.name "Your Personal Name"
git config user.email "your-personal-email@gmail.com"
  • If it's a work project, run: 
    git config user.name "Your Work Name"
git config user.email "your-work-email@company.com"

Method B: The Automatic includeIf (Recommended)

  1. Organize your projects into folders (e.g., ~/Projects/Personal and ~/Projects/Work).
  2. Set your *default* identity as the global one: 
    git config --global user.name "Your Personal Name"
git config --global user.email "your-personal-email@gmail.com"
  3. Create a separate config file for your work identity (e.g., ~/.gitconfig-work). 
    [user]
    name = Your Work Name
    email = your-work-email@company.com
  4. Edit your main ~/.gitconfig to automatically include the work config.
    macOS / Linux 
    [includeIf "gitdir:~/Projects/Work/"]
    path = ~/.gitconfig-work
    Windows (Note: Git config *always* uses forward slashes /) 
    [includeIf "gitdir:C:/Users/YourUserName/Projects/Work/"]
    path = .gitconfig-work

‚ö° Part 3: How to Use Your New Setup (The Workflow)

This is how you use the SSH aliases you created in Part 1.

Cloning a New Repository

You must modify the git clone command to use your new host alias.

  1. Go to GitHub/GitLab and get the SSH clone URL: git@github.com:user/repo.git
  2. Manually change github.com to match your alias.
  • To clone a PERSONAL repo: 
    git clone git@github.com-personal:user/repo.git
  • To clone a WORK repo: 
    git clone git@github.com-work:user/repo.git

Updating an Existing Repository

For any projects you already have, you need to update their "remote" URL.

  1. cd into your project.
  2. Check your current remote: git remote -v
  3. Update the URL to use your alias.
    • For a personal repo: 
      git remote set-url origin git@github.com-personal:user/repo.git
    • For a work repo: 
      git remote set-url origin git@github.com-work:user/repo.git

üîß Part 4: Testing & Troubleshooting

How to Test Your SSH Connection

You can directly test your aliases to see which account they are connecting to.

  • Test Personal Account: 
    ssh -T git@github.com-personal
    Success: Hi <your-personal-username>! You've successfully authenticated...
  • Test Work Account: 
    ssh -T git@github.com-work
    Success: Hi <your-work-username>! You've successfully authenticated...

Common Error: Permission denied (publickey)

This was the error you faced:

git@github.com: Permission denied (publickey).
fatal: Could not read from remote repository.

This error almost always means you are using the wrong URL. You tried to clone using the default git@github.com address, which your ~/.ssh/config file does not have a special rule for.

üí° The Fix: You must use your host alias (e.g., github.com-personal or github.com-work) in your remote URL. Follow the steps in Part 3.
